# 子資源規則

| 層級 | 需要 meta.json？ | 說明 |
|------|----------------|------|
| Actor 目錄 | 是 | 儲存 voice、default_expression 等無法推斷的設定 |
| expressions/ 內的 .png | 否 | 檔名即名稱，存在即有效 |
| bodies/ 內的 .png | 否 | 同上 |
| backgrounds/ 內的圖片 | 否 | 同上 |
| audio/ 內的音檔 | 否 | 同上 |

# 幀動畫

所有 expression / body 統一視為幀動畫，單張圖就是 1-frame 動畫，不做 Static/Animated 區分。

**幀序列由 filesystem 推論：**
- 掃描目錄內的 .png，依檔名數字排序
- 缺號視為 null（維持上一幀）
- 單張圖 = 序列只有 1 幀

不使用 Sprite Sheet。

**meta.json 儲存無法推論的純量設定：**
```json
{
  "fps": 12,
  "default_expression": "普通"
}
```
meta.json 由 App in-app 編輯器維護，只處理簡單數值設定，**不索引子資源路徑**（子資源路徑永遠靠掃描取得）。

# 掃描與匯入

**掃描（Refresh）**
App 啟動或使用者手動觸發。掃描 assets/ 目錄：
- 發現 actor 目錄沒有 meta.json → 自動建立預設 meta.json
- 發現 expressions/、bodies/ → 列舉所有 .png，無需任何設定

**匯入 UI（未來）**
本質上是「有導引的掃描」：
1. 幫使用者把素材複製到正確的目錄結構
2. 執行掃描

底層模型與手動放檔案後 Refresh 完全相同。
匯入 UI 是便利功能，不是必要機制，MVP 可先不做。

不使用 FileSystemWatcher（即時監聽 OS 檔案變化）：實作複雜、問題多，觸發式掃描已足夠 MVP。

# 原則
1. 子巢狀資源（script/actor）分開儲存
2. 透過檔案結構索引子巢狀資源
3. meta.json 只記錄無法從檔案結構推斷的 metadata，不記錄子資源路徑
4. Filesystem 是 source of truth；meta.json 補充 filesystem 表達不了的部分

# 決策原因

**1. 使用 JSON 而非 .res/.tres**
`.tres` 的序列化格式把 sub-resource 路徑寫死成 `res://` 或 `uid://`：
```
[ext_resource type="Texture2D" path="res://actors/小傑/sprite.png" id="1"]
```
使用者素材在任意路徑（如 `C:/MyProject/actors/小傑/sprite.png`），這個格式無法表達。

**2. 描述檔不記錄子資源路徑，靠掃描**
記錄路徑 = 兩個 source of truth，檔案改名就壞掉，且沒有 editor 時需要手動維護。
掃描 = filesystem 永遠是唯一真相，描述檔只補充 metadata。

**3. 不使用 Sprite Sheet**
創作者工作流不需要（直接儲存單張圖）。幀序列靠檔名排序推論，單張圖即 1-frame，不需要特殊處理。
