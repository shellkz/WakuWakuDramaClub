# 劇本編輯器自動補全

## 前提

資源由 filesystem 主宰，`ResourceManager` 掃描使用者專案的 `assets/` 後建立 `AssetIndex`。

Editor 不直接掃 filesystem，而是透過 `ResourceManager` 取得候選資料。

目前資源 ID 採用 namespace 格式：

```text
pack/name
```

例如：

```text
default/毛豆
default/文藝社團 早上
default/音效 問號
```

## 想達成的效果

補全的目標是讓使用者在寫劇本時，能依照目前所在的語法位置看到合理候選，而不是只依靠字串 prefix 猜測。

使用者體驗上，希望可以做到：

```text
打字
顯示 instruction 候選
選擇 instruction
顯示 instruction 需要的參數候選
選擇參數
顯示這個 instruction 可用的 statement 候選
選擇 statement
顯示這個 statement 需要的參數候選
重複直到這行完成
```

補全候選由目前游標所在的語法位置決定：

```text
正在輸入主要 instruction      -> 補 instruction keyword / actor id
正在輸入主要 instruction 參數 -> 補該 instruction 需要的主資源或參數
正在輸入 option statement     -> 補目前 instruction 可用的 statement keyword
正在輸入 statement 參數       -> 補該 statement 需要的參數
正在輸入對話文字              -> 不顯示 options
```
## 提供候選種類
### 指令 keyword

例如：

```text
背景
音效
登場
移動
等待
```

### 指令選項 keyword

```text

登場 的 在
移動 的 從、到
```

### 指令的參數


```text
對話 的 default/毛豆             # Actor
背景 的 default/文藝社團 早上     # Background
音效 的 default/音效 問號         # Audio
```

### 指令選項的參數

```text
default/毛豆 "對話" 的 表情  # Actor.Expressions
```

## 需求案例

行首或第一個 token：

```text
[cursor]     -> 補 instruction keyword + actor id
def[cursor]  -> 補可接續的 actor id / resource id
```

背景：

```text
背景 [cursor]    -> 補 background id
背景 def[cursor] -> 補符合 prefix 的 background id
```

音效：

```text
音效 [cursor]    -> 補 audio id
音效 def[cursor] -> 補符合 prefix 的 audio id
```

Actor 後方：

```text
default/毛豆 [cursor]
```

補這個 actor 可以做的主要行為：

```text
對話
登場
移動
```

選 `對話` 時，插入空對話並把 caret 移到引號中間：

```text
default/毛豆 "[cursor]"
```

對話文字中：

```text
default/毛豆 "你[cursor]好"
```

不顯示 option statement 候選。

對話後方：

```text
default/毛豆 "你好" [cursor]
```

補對話可用的 option statements，例如：

```text
表情
動作
```

對話的表情：

```text
default/毛豆 "你好" 表情 [cursor]
```

補該 actor 可用的 expressions：

```text
高興
發呆
```

Actor 移動：

```text
default/毛豆 移動 [cursor]
```

補移動可用的 option statements，例如：

```text
到
在
```

移動目的地：

```text
default/毛豆 移動 到 [cursor]
```

補站位：

```text
左
中
右
```

## Parser 與 Line Analyzer

補全不直接使用完整的 `ScriptParser.Parse()` 解讀目前行。

正式 parser 的責任是解析完整劇本，建立 `Instruction`；但補全發生時，使用者常常正在輸入半句：

```text
背景 default/
default/毛豆 "台詞" 表
default/毛豆 移動 到 
```

這些狀態對正式 parser 可能是 invalid，但對補全來說是正常狀態。

因此補全應使用寬鬆的 line analyzer：

```text
ScriptParser
  解析完整劇本，建立 Instruction

ScriptLineAnalyzer / CompletionLineAnalyzer
  寬鬆分析目前行、游標位置、目前 token、目前 statement
```

兩者可以共用底層 tokenizer / keyword 定義：

```text
ScriptTokenizer.TokenizeLine(line)
ScriptParser.GetSupportedKeywords()
```

原則：

```text
不要讓 completion 跑完整 Parse()
可以讓 completion 和 parser 共用 tokenizer
```

## 前置作業

### Instruction schema

completion provider 需要知道：

```text
有哪些 instruction
每個 instruction 的 primary statement 需要哪些 arguments
每個 instruction 支援哪些 option statements
每個 option statement 需要哪些 arguments
每個 argument 的候選來源是什麼
```

目前 parser 已經會掃描所有 `Instruction` 子類別，建立 instruction name 到 instruction type 的 factory。
但這個 factory 只知道：

```text
instruction name
instruction C# type
```

它不知道 instruction 支援哪些 statement 結構。

現況中，這些知識隱含在各個 instruction 的 `BindData()` 裡：

```text
DialogueInstruction
  讀取 primary statement: 對話 [speaker, speech]
  讀取 option statement: 表情 [expression]
  讀取 option statement: 動作 [pose]

ActorMoveInstruction
  讀取 primary statement: 移動 [actor]
  讀取 option statement: 到 [destination]
  讀取 option statement: 在 [duration]
```

人可以從 `BindData()` 看出 schema，但程式無法穩定地從 `BindData()` 推導 schema。

實作採用折衷方式：

```text
讓 schema 和 BindData 共用同一批 declarative binding definitions。
不要從 BindData 執行結果反推 schema。
```

也就是不要讓 `BindData()` 本身偷偷產生 schema。
`BindData()` 仍然只負責把 `RawInstruction` 的資料綁到 runtime 欄位。

建議方向：

```text
Instruction class
  定義 declarative binding definitions / schema
  BindData(raw) 使用同一份 definitions 取值
  CompletionProvider 使用同一份 definitions 產生候選
```

概念上：

```text
InstructionBindingDefinition
  InstructionName
  PrimaryStatement
  OptionStatements

StatementBindingDefinition
  StatementName
  Arguments

ArgumentBindingDefinition
  Name
  ValueKind
  Required / Optional
```

`ValueKind` 用來描述候選來源，例如：

```text
Actor
DialogueText
Background
Audio
ActorExpression
ActorPose
Anchor
Duration
```

例如對話 instruction 的 definitions 可以描述成：

```text
Instruction: 對話
PrimaryStatement:
  對話
	actor: Actor
	speech: DialogueText
OptionStatements:
  表情
	expression: ActorExpression
  動作
	pose: ActorPose
```

移動 instruction 的 definitions 可以描述成：

```text
Instruction: 移動
PrimaryStatement:
  移動
	actor: Actor
OptionStatements:
  到
	destination: Anchor
  在
	duration: Duration
```

### InstructionRegistry

建立一個 `InstructionRegistry`，負責掃描所有 `Instruction` 子類別，集中保存：

```text
instruction name -> instruction type
instruction name -> instruction schema / binding definitions
all instruction keywords
all option statement keywords
```

責任分工：

```text
InstructionRegistry
  語法與 instruction 能力的 registry。

ResourceManager
  資源與素材候選，例如 actor、background、audio、actor expression。

CompletionProvider
  根據 CompletionLineResult + InstructionRegistry + ResourceManager 產生候選。
```

避免把 instruction schema 放進 `ResourceManager`。
`ResourceManager` 應只負責資源索引；instruction schema 是劇本語言本身的定義。

原本 hardcoded 的 global statement keyword：

```text
在
到
表情
動作
```

未來應改由 `InstructionRegistry` 從各 instruction schema 收集。

### RawInstruction-like line model

指令本質上是一群 statement：

```text
statement statement statement...
```

第一個 statement 代表主要 instruction，後面的 statement 代表 options / modifiers。
每個 statement 由 type 與多個以空白分隔的 arguments 組成：

```text
type arg arg arg...
```

角色指令為了書寫方便，會打破「第一個 token 是 instruction keyword」的規則：

```text
default/毛豆 "台詞"
default/毛豆 登場
default/毛豆 移動 到 右
```

但 parser 在建立 `RawInstruction` 前已經會把這些 shorthand normalize 回一般結構：

```text
default/毛豆 "台詞"
  -> 對話 default/毛豆 台詞

default/毛豆 登場
  -> 登場 default/毛豆

default/毛豆 移動 到 右
  -> 移動 default/毛豆 到 右
```

因此補全可以從 `RawInstruction` 的概念下手：先把目前行寬鬆分析成一個 RawInstruction-like 物件，再依照游標所在位置決定候選。

但補全不應直接使用正式 `RawInstruction` 作為唯一資料結構，因為補全時目前行經常是不完整的：

```text
背景 default/
default/毛豆 "台詞" 表
default/毛豆 移動 到 
```

建議另外建立 editor 專用的 line model，例如：

```text
CompletionLineResult
  Raw: CompletionRawInstruction
  Cursor: CompletionCursorContext

CompletionRawInstruction
  Type
  Statements
  IsActorShorthand

CompletionStatement
  Type
  Arguments
  StartColumn
  EndColumn
```

正式 `RawInstruction` 偏向「已可建立 Instruction 的中介資料」；`CompletionRawInstruction` 則偏向「目前這行即使尚未合法，也能描述到什麼程度」。

#### Primary statement 與 option statements

`RawInstruction.Type` 由 primary statement 決定。

primary statement 是 normalize 後 statement list 的第一個 statement；後續 statements 則是 option / modifier statements。

```text
CompletionRawInstruction.Type = Statements[0].Type
Statements[0]                 = primary statement
Statements[1..]               = option statements
```

例如：

```text
背景 default/文藝社團 早上
```

會被理解成：

```text
Type = 背景
Statements[0] = 背景 [default/文藝社團, 早上]
```

角色 shorthand normalize 後也遵守同一條規則：

```text
default/毛豆 移動 到 右
```

會被理解成：

```text
Type = 移動
Statements[0] = 移動 [default/毛豆]
Statements[1] = 到 [右]
```

因此 completion provider 不需要依賴原始文字判斷「這行是不是角色開頭」，只要看 normalize 後的結構：

```text
Raw.Type
Statements[0]
Statements[1..]
```

正式 `RawInstruction.Statements` 目前使用 `Dictionary<string, Statement>`，方便 instruction bind data 時用 keyword 查詢，例如 `表情`、`動作`、`到`。

但 editor line model 需要保留 statement 順序與游標位置，因此 `CompletionRawInstruction.Statements` 應使用 `List<CompletionStatement>`。是否為 primary statement 可由 index 判斷，不一定要額外存 `IsPrimary`：

```text
statementIndex == 0  -> primary statement
statementIndex > 0   -> option statement
```

補全流程：

```text
目前行 + caret column
  -> tokenize with position
  -> normalize actor shorthand
  -> group into statements
  -> build RawInstruction-like line model
  -> locate cursor context
  -> completion provider 根據 context 給候選
```

```
InstructionSchema
  Name
  PrimaryStatement: StatementSchema
  OptionStatements: List<StatementSchema>

StatementSchema
  Name
  Arguments: List<ArgumentSchema>

ArgumentSchema
  Name
  ValueKind
  Required
  DefaultValue?
```


### 實作步驟

導入新的 completion 架構時，實作順序：

```text
0. 完成前置作業
   定義 instruction schema / declarative binding definitions
   建立 InstructionRegistry
   讓 parser 與 completion 共用 registry

1. 定義 completion 資料結構
   CompletionLineResult
   CompletionRawInstruction
   CompletionStatement
   CompletionCursorContext
   CompletionOption

2. 實作 CompletionAnalyzer
   input: line text + caret column
   output: CompletionLineResult

3. 實作 CompletionProvider
   input: CompletionLineResult + resources / instruction schema
   output: CompletionOption list

4. UI 層接 CodeEdit API
   input: CompletionOption list
   action: AddCodeCompletionOption(...) + UpdateCodeCompletionOptions(true)
```

責任分工：

```text
CompletionAnalyzer
  理解目前這行的語法結構，以及游標正在填哪個 slot。

CompletionProvider
  根據 CompletionLineResult 決定這個 slot 有哪些候選。

EditingMenu / ScriptEditor UI
  只負責接 CodeEdit event、取得目前行與 caret column、顯示候選。
```

UI 層不應再自己判斷「現在是不是背景、音效、actor 後方」。
這些語法判斷應集中在 `CompletionAnalyzer`，候選判斷應集中在 `CompletionProvider`。

`CompletionCursorContext` 是提供候選的唯一語法依據，用來描述游標正在填哪個 slot。

它至少要能分辨：

```text
InstructionName
InstructionArgument
StatementName
StatementArgument
DialogueText
Unknown
```

completion provider 應該只依賴 `CompletionLineResult.Raw` 與 `CompletionLineResult.Cursor` 產生候選。
舊式的淺層 `CompletionContext` enum 不再需要保留，應由 `CompletionCursorContext` 取代。

這樣 actor shorthand 的特例只存在於 normalize / analyze 階段；completion provider 不需要到處判斷「這行是不是角色開頭」，只要看被整理後的 instruction structure。

## Godot CodeEdit 實作注意

目前先使用 Godot `CodeEdit` 內建 code completion。

### API 使用方式

啟用補全：

```text
ScriptEditor.CodeCompletionEnabled = true
```

監聽文字變更，當目前狀態需要補全時，主動要求 CodeEdit 重新開啟補全：

```text
ScriptEditor.TextChanged += OnScriptChanged
ScriptEditor.RequestCodeCompletion(true)
```

新的觸發 request 規則應該直接使用 analyzer/provider：

```text
TextChanged
  -> 取得目前行 + caret column
  -> CompletionAnalyzer.Analyze(line, caretColumn)
  -> CompletionProvider.GetOptions(result)
  -> options.Count > 0 才 RequestCodeCompletion(true)
```

也就是：

```text
trigger completion = provider 對目前 CompletionLineResult 有候選
```

`ShouldRequestCompletion()` 不應再另外用字串規則判斷 context，否則「要不要跳補全」和「實際能補什麼」會變成兩套邏輯。

概念上：

```text
ShouldRequestCompletion:
  result = analyzer.Analyze(currentLine, caretColumn)
  options = provider.GetOptions(result)
  return options.Count > 0

OnCodeCompletionRequested:
  result = analyzer.Analyze(currentLine, caretColumn)
  options = provider.GetOptions(result)
  add options to CodeEdit
```

如果 `ShouldRequestCompletion()` 和 `OnCodeCompletionRequested()` 各自呼叫一次 analyzer/provider，同一行與同一 caret 位置必須得到相同候選。
第一版可以先不 cache；若之後需要，再 cache 最近一次的 line、caret column、result、options。

監聽補全請求，在 callback 內加入候選：

```text
ScriptEditor.CodeCompletionRequested += OnCodeCompletionRequested
ScriptEditor.AddCodeCompletionOption(kind, displayText, insertText, color)
ScriptEditor.UpdateCodeCompletionOptions(true)
```

監聽候選確認，用於特殊插入行為。例如選擇對話時插入 `""` 後，把 caret 移回兩個引號中間：

```text
ScriptEditor.CodeCompletionOptionConfirmed += OnCodeCompletionOptionConfirmed
ScriptEditor.SetCaretColumn(...)
```

實測行為：

- `CodeEdit.CodeCompletionRequested` 觸發後，在 handler 內呼叫 `AddCodeCompletionOption()` 與 `UpdateCodeCompletionOptions(true)`。
- `UpdateCodeCompletionOptions()` 會依 `GetTextForCodeCompletion()` 的 query text 過濾候選。
- 如果 query text 是空字串，或 query text 與候選文字不匹配，候選可能會被過濾到 0，popup 不會顯示。
- 因此 `CodeEdit` 內建 completion 較適合 prefix-based completion，例如輸入 `def` 後補 `default/毛豆`。
- `CodeCompletionOptionConfirmed` 可用來處理選項被套用後的 caret 位置調整。

這代表目前先以「使用者已輸入 prefix」的補全為主：

```text
def          -> default/毛豆
背景 def     -> default/文藝社團 早上
音效 def     -> default/音效 問號
```

若未來要支援「空行或行首直接顯示全部候選」，可能需要另外實作自訂 popup/list UI，而不是完全依賴 `CodeEdit` 內建 completion。

## 暫緩項目

### 錯誤提示 / Hover / 跳轉素材

有了 `AssetRecord.RelativePath` 後可實作：

- 找不到 `pack/name` 時顯示錯誤。
- hover background 顯示背景預覽。
- hover audio 顯示音檔資訊，未來可播放。
- hover actor 顯示 actor body / default expression。
- hover expression 時，依目前 actor 找到 expression png 預覽。
- 跳轉素材位置。

## 設計原則

- Editor 不直接掃 filesystem。
- `AssetIndex` 是補全候選的基礎。
- 補全不直接呼叫完整 `ScriptParser.Parse()`。
- 補全使用寬鬆 line analyzer；正式 parser 和 line analyzer 可共用 tokenizer / keyword 定義。
- 主資源使用 `pack/name`。
- 子資源名稱是 actor-local，例如 `高興`，不需要寫成 `default/毛豆/高興`。
- `DialoguePanel` 顯示名稱使用 `ActorDefinition.DisplayName`，不是 script 裡的完整 actor id。
