# 劇本自動補全

自動補全的目標是讓使用者在寫劇本時，依照目前所在的語法位置看到合理候選，而不是靠記憶或只用字串 prefix 猜測。

補全系統只負責輔助輸入，不負責執行劇本。完整劇本仍由 parser 解析成 instruction，再交給 timeline / renderer。

## 基本原則

- Editor 不直接掃 filesystem。
- 資源候選一律由 `ResourceManager` 提供。
- 指令與選項結構一律由 instruction schema / registry 提供。
- 補全不直接呼叫完整 parser。
- 補全使用寬鬆的單行分析器，允許目前行尚未完成。
- UI 層只負責顯示候選，不直接判斷劇本語法。

## 資源 ID

素材由專案的 `assets/` 目錄提供，資源 ID 使用 `pack/name` 格式：

```text
default/毛豆
default/文藝社團 早上
default/問號
```

角色的子資源，例如表情與動作，使用 actor-local 名稱：

```text
高興
發呆
mic
```

因此劇本中不需要把表情寫成完整路徑：

```text
default/毛豆 「你好」 表情 高興
```

## 補全流程

使用者輸入一行劇本時，補全系統會做以下處理：

```text
目前行 + caret column
  -> tokenize with position
  -> normalize actor shorthand
  -> group into statements
  -> build completion line model
  -> locate cursor context
  -> provide candidates
```

正式 parser 只處理完整劇本；補全 analyzer 要能理解不完整的輸入：

```text
背景 default/
default/毛豆 「台詞」 表
default/毛豆 移動 到
```

這些對完整 parser 可能還不是合法劇本，但對補全來說是正常輸入狀態。

## 語法模型

一行劇本會被理解成一個 primary statement 加上多個 option statements：

```text
primary statement option statement option statement...
```

每個 statement 都由 keyword 與 arguments 組成：

```text
keyword argument argument...
```

例如：

```text
對話 default/毛豆 「你好」 表情 高興 動作 mic
```

可理解成：

```text
primary:
  對話 [default/毛豆, 你好]

options:
  表情 [高興]
  動作 [mic]
```

角色 shorthand 會先 normalize 成正式結構：

```text
default/毛豆 「你好」
  -> 對話 default/毛豆 「你好」

default/毛豆 登場 在 左
  -> 登場 default/毛豆 在 左

default/毛豆 移動 從 左 到 右
  -> 移動 default/毛豆 從 左 到 右
```

補全 provider 只需要看 normalize 後的結構，不需要到處特判「這行是不是角色開頭」。

## 游標位置

補全候選由游標所在的語法位置決定。

```text
行首或第一個 token        -> instruction keyword / actor id
主要 instruction 參數     -> 該 instruction 的參數候選
actor 後方                -> 對話 snippet / actor action
option statement keyword  -> 目前 instruction 可用的 option
option statement 參數     -> 該 option 的參數候選
對話文字中                -> 不顯示候選
```

範例：

```text
[cursor]
```

候選：

```text
背景
音效
等待
對話
登場
移動
default/毛豆
```

```text
背景 [cursor]
```

候選：

```text
default/文藝社團 早上
default/赤冬 會長辦公室 晚上
```

```text
default/毛豆 [cursor]
```

候選：

```text
「」
登場
移動
```

```text
default/毛豆 「你好」 表情 [cursor]
```

候選：

```text
高興
發呆
angry
```

```text
default/毛豆 移動 到 [cursor]
```

候選：

```text
左
中
右
```

## 候選來源

候選來源由 argument 的 value kind 決定。

```text
Actor
  ResourceManager.GetActorIds()

Background
  ResourceManager.GetBackgroundIds()

Audio
  ResourceManager.GetAudioIds()

ActorExpression
  依目前 primary actor 查 actor expressions

ActorPose
  依目前 primary actor 查 actor bodies / poses

Anchor
  左 / 中 / 右

Float
  第一版不提供候選，保留給 validation 使用

DialogueText
  不提供候選
```

## 指令選項

每個 instruction 由 schema 定義可用的 option statements。

例如 `對話`：

```text
Primary:
  對話 actor dialogue

Options:
  表情 expression
  動作 pose
```

例如 `登場`：

```text
Primary:
  登場 actor

Options:
  在 anchor
```

例如 `移動`：

```text
Primary:
  移動 actor

Options:
  從 anchor
  到 anchor
  在 duration
```

`到` 是 `移動` 的必要 option；`從` 與 `在` 是可選 option。

## Option Statement 順序

Primary statement 必須出現在第一個 statement。

Option statements 可以任意順序出現，因為它們有自己的 keyword：

```text
default/毛豆 移動 從 左 到 右
default/毛豆 移動 到 右 從 左
```

兩者語意相同。

但缺少必要 option 時應在 validation / diagnostics 階段報錯：

```text
default/毛豆 移動 從 左
```

這行缺少 `到`。

## 對話引號

劇本對話使用全形引號：

```text
default/毛豆 「你好」
```

不使用半形雙引號：

```text
default/毛豆 "你好"
```

原因是 editor 的 code completion 與字串 token 行為容易受 `"` 影響。劇本語言統一由 `LanguageSchema` 定義對話 quote：

```text
DialogueQuoteStart = 「
DialogueQuoteEnd   = 」
EmptyDialogueText  = 「」
```

選擇對話 snippet 時，補全會插入：

```text
「」
```

並把 caret 移到兩個引號中間。

## 責任分工

```text
InstructionRegistry
  管理 instruction keyword、option keyword、schema。

ResourceManager
  管理 actor、background、audio、actor expression、actor pose 等資源候選。

CompletionAnalyzer
  分析目前行與 caret，判斷游標正在填哪個語法 slot。

CompletionProvider
  根據 analyzer 結果、schema、resources 產生候選。

EditingMenu / ScriptEditor
  接 editor event，顯示候選，處理選項確認後的 caret 行為。
```

UI 層不應直接判斷「現在是不是背景、音效、actor 後方」。

## 設計原則

- instruction schema 是劇本語言的定義。
- resource manager 是素材索引，不保存語法規則。
- analyzer 可以接受不完整輸入。
- provider 只根據 cursor context 產生候選。
- actor shorthand 的複雜度只留在 normalize / analyze 階段。
- 補全文件只描述整體運作概念；細部 API 行為記錄在開發筆記。
